# Companion Agent — Architecture
**Spraying work order · AGRIVI 360 Farm Enterprise**

*Design thesis: the LLM is not the agent — the reducer is. Every turn crosses one seam, `utterance → Propose → Verify → Commit`, where the proposer (a small extraction model — Gemini 2.5 Flash-Lite here, or an on-device parser when offline) emits raw text only and the verifier alone mints ids — so the model is a swappable implementation of a fixed contract, and a dead cell costs fluency, nothing else.*

### 1. State management

State is not the transcript — it is `fold(events)` over an append-only log, reduced by a pure function. `PROPOSAL_RECEIVED` records a *claim* and deliberately mutates nothing; only `SLOT_COMMITTED`, emitted exclusively by the verifier, changes a slot, and each slot carries `{status, value, raw, provenance}` so the audit trail is a property of the design, not a bolted-on log (crash recovery and replay fall out for free). **Two fields in one message is the normal case, not an edge case**: workers speak in bursts, so the extractor's strict-schema output carries every slot heard and the reducer commits each independently — *"north vineyard, luna, half a litre, this morning"* fills four slots in one turn and the policy asks only for what is still missing. **Ambiguity is a verdict, not a failure**: when candidates score within a threshold, `resolve_field` returns `AMBIGUOUS` with the list rather than guessing, and — critically — the proposer forwards the matched *span* ("the vineyard") rather than resolving it, because resolving in the proposer would destroy the ambiguity before the only component qualified to adjudicate it ever saw it. An explicit crop contradicting the block's season assignment is a distinct case, `CROP_CONFLICT`, which surfaces both facts instead of silently preferring one.

### 2. Tool design

Tools are **pure functions over an on-device mirror of AGRIVI master data**, sharing one signature — `(args) → {verdict: OK | REJECT | AMBIGUOUS, …}` — that never throws and never guesses: `resolve_field` (→ id + derived crop, or candidates, or `UNKNOWN_FIELD`), `resolve_product` (→ PPP id + authorisation no. + label rate), `check_crop_product_compatibility`, `check_dose`, `resolve_operator` (licence valid *on the application date*), and `check_date`. On `REJECT` they return a typed code, a human reason, **and the valid alternatives** — a reject signal must carry enough to drive a repair turn, or the agent can only say "invalid". The load-bearing rule is that **the model's opinion about validity is never trusted**: the verifier re-runs the whole suite on every commit regardless of what the proposer asserted, so the LLM needs no tool call to *validate* — the catalogue ships in a cached system prefix and one strict JSON-schema-constrained `propose_slots` call answers the turn, because on a 2G cell every extra round trip multiplies the failure surface. The one genuinely agentic tool, `lookup_product_label`, exists for when the worker asks a question mid-flow. Same functions, two callers: advisory when the model reads them, authoritative when the reducer runs them.

### 3. One failure mode — the write that might have happened

**The trigger** is a submit that goes out and never answers — timeout, dead cell, tunnel — leaving the client unable to distinguish *"the write never landed"* from *"it landed and the ACK was lost."* **The agent never blind-retries**: the record enters a durable **outbox** before the network is touched, keyed by an **idempotency key derived from a content hash**; retries replay the *same* key under exponential backoff, the server treats it as a uniqueness constraint and returns the original outcome, and the worker is told **"queued", never "saved"**, until an ACK arrives (a genuine correction hashes differently and is correctly a new write). **I chose this one** because a naive retry writes the spray twice, and under Reg. (EU) 1107/2009 Art. 67 a duplicate is *worse* than a missing record — a missing one can be re-entered from memory, but a phantom second application makes the block look over-dosed on paper, failing an audit and breaking MRL traceability. That asymmetry is the argument: this is the failure where the obvious response silently manufactures a compliance defect, so it is designed for rather than caught. The consequence is that slots, tools, validation and policy are all local and deterministic and only the proposer depends on the network — so a worker in a dead zone still finishes the log against a constrained parse, it syncs on reconnect, and it can never be written twice.
