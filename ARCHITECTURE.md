# Companion Agent — Architecture
**Spraying work order · AGRIVI 360 Farm Enterprise**

**The LLM is not the agent — the reducer is.** Every turn crosses one seam, the *stochastic–deterministic boundary*: `utterance → Proposer → Verifier → Commit`, with a typed `Reject` back-edge. The proposer (Claude, or an on-device parser) emits **raw text only** — it never sees an AGRIVI id, never writes state, never decides what to ask. Only the verifier mints ids. This one rule is both the safety argument and the connectivity answer: the model is a swappable implementation of a fixed contract, not the agent, so a dead cell costs fluency and nothing else.

### 1. State management

State is not the transcript — it is `fold(events)` over an append-only log, reduced by a pure function. `PROPOSAL_RECEIVED` records a *claim* and deliberately mutates nothing; only `SLOT_COMMITTED`, emitted exclusively by the verifier, changes a slot. Each slot carries `{status, value, raw, provenance}`, provenance naming who proposed it and which tool verified it — so the audit trail is a property of the design, not a bolted-on log. Crash recovery, time-travel replay, and a determinism check (fold twice, compare) fall out for free. **Two fields in one message is the normal case**, not an edge case: workers speak in bursts, so the envelope carries every slot heard and the reducer commits them independently — *"north vineyard, luna, half a litre, this morning"* fills four slots in one turn and the policy asks only for what's missing. **Ambiguity is a verdict, not a failure**: when candidates score within a threshold, `resolve_field` returns `AMBIGUOUS` with the list rather than guessing. Critically, the proposer forwards the matched *span* ("the vineyard") rather than resolving it itself — resolving in the proposer would destroy the ambiguity before the only component qualified to adjudicate it ever saw it. Conflict is distinct: an explicit crop contradicting the block's season assignment yields `CROP_CONFLICT`, surfacing both facts instead of silently preferring one.

### 2. Tool design

Tools are **pure functions over an on-device mirror of AGRIVI master data**, with one signature — `(args) → {verdict: OK | REJECT | AMBIGUOUS, …}` — that never throws and never guesses: `resolve_field` (→ id + derived crop, or candidates, or `UNKNOWN_FIELD`), `resolve_product` (→ PPP id + authorisation no. + label rate), `check_crop_product_compatibility`, `check_dose`, `resolve_operator` (licence valid *on the application date*), `check_date`. On `REJECT` they return a typed code, a human reason, **and the valid alternatives** — a reject signal must carry enough to drive a repair turn, or the agent can only say "invalid". **Crop is derived, never asked**: AGRIVI assigns crop to the *block* for the season, so `resolve_field` returns it and a required slot disappears. The load-bearing claim: **the model's opinion about validity is never trusted.** The verifier re-runs the suite on every commit regardless of what any proposer asserted — the LLM's job is reading, not judging. So the model needs no tool call to *validate*: the catalogue ships in a cached system prefix and one strict-schema `propose_slots` call answers the turn, because on a 2G cell every extra round trip multiplies the failure surface. It gets one genuinely agentic tool, `lookup_product_label`, for when the worker asks a question mid-flow. Same functions, two callers: advisory when the model reads them, authoritative when the reducer runs them.

### 3. One failure mode: the write that might have happened

**Trigger.** Submit goes out; the answer never comes back — timeout, dead cell, tunnel. The client cannot distinguish *"the write never landed"* from *"it landed and the ACK was lost."*

**Response.** Never blind-retry. The record enters a durable **outbox** before the network is touched, keyed by an **idempotency key derived from a content hash**. Retries replay the *same* key under exponential backoff; the server treats it as a uniqueness constraint and a replay returns the original outcome. The worker is told **"queued", never "saved"**, until an ACK arrives. A genuine correction hashes differently and is correctly a new write.

**Why this one.** A naive retry writes the spray twice, and in a record mandated by Reg. (EU) 1107/2009 Art. 67 a duplicate is **worse than a missing record**: a missing one can be re-entered from memory, but a phantom second application makes the block look over-dosed on paper — failing an audit, breaking MRL traceability, potentially condemning the harvest. That asymmetry is the argument. This is the failure where the obvious response ("just try again") silently manufactures a compliance defect, so it gets designed for rather than caught. Under simulated flaky signal the implementation commits the write *and then* drops the ACK; the outbox retries and the server returns the original record. The tests assert the counterfactual too: a client regenerating its key produces two records.

**Consequence.** Slots, tools, validation and policy are all local and deterministic; the only network-dependent component is the proposer, replaceable at runtime. Offline, because the policy asks **one slot at a time**, an open-domain parse collapses into a *constrained* parse against a known catalogue — so the offline path degrades in fluency, not in capability. The machine owns every question and the model contributes only the acknowledgement, so the agent asks identical questions online and off. A worker in a dead zone finishes the log; it syncs on reconnect; it can never be written twice.
